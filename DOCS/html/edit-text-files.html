Concerning language packs and the third party program Editarea
In the js.js file, Editarea is sent the instruction to read a cookie from the CSS editor, containing the language initials, with javascript. The obvious downside of this, is that, when you add more languages in the CSS editor GUI template, the Editarea program also must contain language packs for those languages.

Relevant configuration options:
Settings menu > Core config > Structure menu > global > Editor menu > lang_cookie
Settings menu > Editor core-client (or * handling) > Structure menu >
extlink_gui_lang > Editor menu * defaults to gui which is a css class in the language database (css) file.
ext_links > Editor menu

Edit relevant files:
Settings menu > Edit button left to flag of the language set you're editing > Structure menu > gui (or create a new)



version 0.1.8 and earlier: Unless the file you edit was requested with a fd urlquery containing the string "lang", the current selected language it's cssfilter will be applied on the contents of the editarea provided that there has been a valid load of a set of translation instructions using the extlink_lang configuration directive, resulting in recognized foreign language conversion to english when the file is actually saved to disk.

Relevant configuration options:

Settings menu > Editor core-client (or * handling) > Structure menu >
global > translate_editfield (option introduced in 0.1.9)
extlink_lang > Editor menu * defaults css files to "css" which is a css class in the language database (css) file.
ext_links > Editor menu

Edit relevant files:

Settings menu > Edit button left to flag of the language set you're editing > Structure menu > css - not available for default english pack (or create a new)
